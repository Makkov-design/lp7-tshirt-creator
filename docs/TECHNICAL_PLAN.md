# Technical Plan: LP-7 T-Shirt Creator

## 1. Рекомендованный стек

Рекомендация для MVP:
- Next.js App Router + TypeScript
- Tailwind CSS
- Motion for React
- Supabase Postgres + Realtime
- Vercel
- Zod для валидации формы
- Fuse.js для fuzzy-поиска участника
- React Hook Form или легкий локальный state, если форма останется короткой

Почему этот стек:
- Next.js хорошо подходит для полноценного React-приложения с API-роутами и быстрым деплоем.
- Tailwind ускоряет создание темного кастомного UI без тяжелого дизайн-кита.
- Motion for React закрывает плавные переходы между шагами и финальную анимацию.
- Supabase дает Postgres, простую таблицу, realtime-обновления и экспорт данных.
- Vercel дает быстрый деплой Next.js-проекта без отдельного сервера.
- Fuse.js подходит для поиска по маленькому списку из 29 участников с опечатками и разными вариантами написания.

Официальные источники:
- Next.js docs: https://nextjs.org/docs
- Tailwind CSS docs: https://tailwindcss.com/docs
- Motion for React docs: https://motion.dev/docs/react
- Supabase docs: https://supabase.com/docs
- Vercel Next.js docs: https://vercel.com/docs/frameworks/full-stack/nextjs
- Fuse.js docs: https://www.fusejs.io/

## 2. Альтернативы

### Более простой вариант

Vite + React + Supabase + Vercel.

Плюсы:
- проще структура;
- быстрее старт;
- меньше серверных концепций.

Минусы:
- если позже понадобится админка, API, server actions или защищенные операции, Next.js будет удобнее.

### Более строгий вариант на будущее

Next.js + Supabase Auth или invite-токены.

Плюсы:
- почти исключает дубли;
- можно заранее привязать каждого участника к ссылке.

Минусы:
- чуть больше подготовки перед запуском;
- нужно собрать список участников заранее.

Текущее решение: invite-ссылки не используем, потому что ссылка будет одна для общего чата.

## 3. Архитектура MVP

```text
Browser
  -> Next.js app
    -> form state and animated flow
    -> t-shirt preview component
    -> Supabase client for read/realtime grid
    -> server action or route handler for submission
  -> Supabase Postgres
    -> submissions table
```

## 3.1 Mobile-First Implementation Rules

- Верстка проектируется от мобильного viewport вверх.
- Базовая ширина для ручной проверки: 360px и 390px.
- Desktop-брейкпоинты добавляются после того, как мобильный флоу стабилен.
- Основной мокап и текущий шаг формы должны быть видны без хаотичного скролла.
- Touch targets для кнопок, размеров и карточек участников должны быть не меньше 44px по высоте.
- Проверять состояния с открытой мобильной клавиатурой для шагов имени и 3 слов.
- Не использовать hover-only взаимодействия как обязательные для прохождения флоу.

## 6. Дизайн мокапа

MVP-подход:
- нарисовать футболку CSS/SVG как компонент;
- накладывать текстовые зоны поверх;
- обеспечить адаптивный масштаб;
- ограничить длину вводимых значений.

Более качественный вариант:
- использовать растровый мокап футболки PNG/WebP;
- наложить текст поверх через HTML/CSS;
- позже подготовить отдельные экспортные мокапы.

Рекомендация: начать с CSS/SVG-мокапа, если нет готового изображения футболки. Если есть реальный мокап от производителя, использовать его.

## 7. База данных

Минимальные таблицы:

```sql
create table public.participants (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  first_name text not null,
  last_name text not null,
  search_name text not null,
  photo_url text,
  photo_file_name text,
  status text not null default 'waiting'
);
```

```sql
create table public.submissions (
  id uuid primary key default gen_random_uuid(),
  created_at timestamptz not null default now(),
  participant_id uuid not null references public.participants(id),
  first_name text not null,
  last_name text not null,
  size text not null,
  word_1 text not null,
  word_2 text not null,
  word_3 text not null,
  client_submission_id text
);
```

Индексы/ограничения:

```sql
create unique index submissions_participant_id_idx
on public.submissions (participant_id);
```

```sql
create unique index submissions_client_submission_id_idx
on public.submissions (client_submission_id)
where client_submission_id is not null;
```

Для поиска:

```sql
create index participants_search_name_idx
on public.participants (search_name);
```

## 8. Защита от дублей

MVP:
- участник находит себя в списке по имени/фамилии;
- сайт показывает фото и просит подтвердить "это я";
- перед открытием формы проверяется, нет ли заявки для `participant_id`;
- база запрещает повторную заявку через unique index по `participant_id`;
- после успешной отправки `participants.status` меняется на `submitted`;
- `localStorage` хранит последний `participantId` и `submissionId`, чтобы тот же браузер сразу видел финальную сцену.

Важно:
- уникальные invite-ссылки не используются;
- одна общая ссылка публикуется в групповом чате;
- защита от случайных дублей надежная, защита от намеренного выбора чужого имени ограниченная.

## 9. Анимации

Минимальный набор:
- `AnimatePresence` для смены шагов;
- layout-анимация обновления мокапа;
- hover/tap-анимации на размерах и кнопках;
- финальный переход футболки в сетку.

Требования:
- не блокировать ввод;
- не делать длительные переходы между каждым маленьким действием;
- учитывать `prefers-reduced-motion`.

## 10. Валидация

Правила:
- участник должен быть выбран из существующего списка;
- нельзя продолжить, если выбранный участник уже имеет заявку;
- размер из разрешенного списка;
- имя и фамилия обязательны;
- каждое слово обязательно;
- максимум 20 символов на слово;
- максимум 24-32 символа на имя/фамилию;
- trim всех строк.

Поиск участника:
- использовать `data/participants.json` как локальный seed-источник;
- нормализовать русский, украинский и английский ввод;
- искать через Fuse.js по `displayName`, `searchName`, aliases и slug;
- при слабой уверенности показывать несколько вариантов вместо автоперехода;
- финальное действие всегда остается ручным: участник подтверждает карточку.

## 11. Роадмап

### Этап 0: Подготовка

Результат:
- утверждены ответы на открытые вопросы;
- подготовлен список участников;
- подготовлены временные плейсхолдеры под фото;
- выбран визуальный референс футболки;
- выбран формат итогового экспорта.

Проверка:
- есть финальный список требований для MVP;
- после исключения "Андрей Троян" и "Віталій Максимук" в seed-данных ровно 29 участников.

### Этап 1: Каркас приложения

Результат:
- создан Next.js-проект;
- подключены Tailwind и базовая тема;
- собрана главная страница;
- есть mobile-first layout.

Проверка:
- приложение запускается локально;
- главная страница сначала проверяется на 360px и 390px viewport;
- desktop viewport проверяется после мобильного.

### Этап 2: Пошаговый флоу

Результат:
- реализованы шаги поиска участника, подтверждения по фото, размера, имени, слов и подтверждения;
- есть валидация;
- можно ходить назад и вперед.

Проверка:
- нельзя продолжить без выбора участника;
- повторный участник видит предупреждение вместо формы;
- невозможно отправить пустые данные;
- валидные данные доходят до финального шага;
- ввод имени и 3 слов удобен при открытой мобильной клавиатуре.

### Этап 3: Мокап футболки

Результат:
- мокап показывает название, девиз, имя, фамилию и 3 слова;
- текст обновляется по мере ввода;
- длинные значения не ломают верстку.

Проверка:
- протестированы короткие и длинные имена/слова;
- мокап читаемый на мобильном экране;
- нет горизонтального скролла на 360px viewport.

### Этап 4: Сохранение и сетка

Результат:
- данные сохраняются в Supabase;
- сетка показывает до 29 слотов;
- после отправки текущий браузер видит финальный экран;
- при обновлении страницы форма не показывается повторно.

Проверка:
- созданная футболка появляется в сетке;
- перезагрузка страницы сохраняет финальный статус.

### Этап 5: Realtime и финальная анимация

Результат:
- сетка обновляется при новых отправках;
- подтвержденная футболка анимированно переходит в сетку.

Проверка:
- две вкладки показывают обновления без ручной перезагрузки.

### Этап 6: Полировка и деплой

Результат:
- финальная dark-theme полировка;
- первичная проверка мобильной версии;
- вторичная проверка desktop-версии;
- деплой на Vercel;
- smoke-test production-ссылки.

Проверка:
- сайт доступен по ссылке;
- отправка работает в production;
- данные доступны для экспорта.

## 12. Предлагаемый порядок реализации

1. Утвердить антидубли через `participant_id` и статус участника в базе.
2. Подготовить список участников и плейсхолдеры под фото.
3. Создать Next.js-проект.
4. Собрать UI без базы на моковых данных.
5. Подключить Supabase и загрузить участников.
6. Добавить поиск участника, фото-подтверждение и блокировку дублей.
7. Добавить сохранение, realtime и финальный статус.
8. Проверить мобильный UX на 360px и 390px viewport.
9. Задеплоить.
